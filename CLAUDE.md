## Project Overview

Sk6.0 is a lottery platform (Web + Android + iOS) built with FastAPI (Python 3.12), PostgreSQL 16, Redis 7, and Celery on the backend, and Flutter for mobile. The entire backend stack runs in Docker Compose. Nginx load-balances across 3 API replicas in production; dev uses a single hot-reloading replica.

## Common Commands

All day-to-day operations go through `make`. Run `make help` to see the full list.

```bash
# Development
make dev              # Start full stack with hot-reload (uvicorn --reload)
make dev-bg           # Same, detached

# Production
make prod / make up   # Start production stack (detached)
make down             # Stop all containers
make restart          # Restart all containers

# Logs & status

make ps               # Container status

# Database migrations (Alembic, runs inside api-1 container)
make migrate          # Apply pending migrations
make migrate-new      # Create new autogenerated revision (prompts for message)
make migrate-down     # Roll back one revision

# Testing

# Linting
make lint             # ruff check + mypy

# Shell access
make shell            # bash inside api-1

# Celery

# Build

```

## Backend Code Architecture

The backend follows **Clean Architecture** (hexagonal/onion). Dependency direction is strictly outer → inner. The domain layer has zero imports from any other layer.

### The Four Layers

| Layer | Location | Contains | Must Never |
|-------|----------|----------|------------|
| Presentation | `app/api/` | FastAPI routers, Pydantic schemas, auth deps, exception handlers | Contain business rules or DB queries |
| Application | `app/application/` | Use-case classes (`CreateRechargeUseCase`), DTOs, transaction coordination | Import FastAPI, HTTP concepts, or SQL |
| Domain | `app/domain/` | Pure Python dataclasses for entities (User, Wallet, Bet), value objects, domain exceptions | Import anything from infrastructure, app, or presentation |
| Infrastructure | `app/infrastructure/` | SQLAlchemy models, repository implementations, Redis client, OTP gateway adapter, Celery tasks | Contain business rules — only persists and fetches |

**Dependency injection** is wired in `app/core/container.py`. FastAPI's `Depends()` resolves concrete implementations at request time, but routers only ever see interfaces defined in `app/application/interfaces/`.

## Database Design

Full DDL is in `backend/migrations/schema.sql`. Key tables:

| Table | Purpose | Key Indexes |
|-------|---------|-------------|
| users | Accounts; status: pending/active/blocked | phone (unique), status, last_login_at |
| wallets | Balance — 1:1 with users, isolated row | user_id (unique) |
| sessions | JWT session tracking for force-logout | token_hash (unique), (user_id, is_active) |
| transactions | Ledger of every money movement | user_id, type, created_at DESC |
| recharge_requests | Recharge lifecycle: pending/approved/rejected | payment_ref (unique where !=rejected), status |
| withdrawal_requests | Withdrawal approvals queue | user_id, status, created_at DESC |
| draws | One row per scheduled draw instance | (schedule_id, draw_date) unique, status |
| bets | User bet tickets | draw_id, ticket_number, user_id |
| results | Published winning numbers | draw_id (unique), published_at |
| winners | Reverse-index from result to winning bets | user_id, draw_id |
| audit_logs | Immutable record of every admin action | actor_id, action, entity_id, created_at |

### Wallet Transaction Safety
All wallet balance changes use three layers of protection — never bypass any of them:

1. **SERIALIZABLE isolation** for all money operations (set on the session/transaction)
2. **`SELECT ... FOR UPDATE`** on the wallet row before any debit — prevents concurrent double-spend
3. **`CHECK (balance >= 0)`** DB constraint — negative balance rejected at the database level
4. Every transaction writes `balance_before` and `balance_after` to the `transactions` table

## Authentication & Authorization

### JWT Design
- Algorithm: HS256, expiry: 24 hours
- Payload fields: `sub` (user UUID), `role` (user/admin/superadmin), `jti` (session ID)
- On login: `jti` is stored in Redis as `session:{jti}` → `{user_id, role, issued_at, status}` with TTL = JWT expiry
- On every request: JWT signature verified, then `jti` looked up in Redis — if missing/revoked → 401

### Roles & RBAC

| Route Pattern | Allowed Roles |
|---------------|---------------|
| `/api/v1/auth/*` | Public |
| `/api/v1/users/me`, `/api/v1/wallet/*` | user, admin, superadmin |
| `/api/v1/bets/*` | user (not blocked) |
| `/api/v1/admin/*` | admin, superadmin |
| `/api/v1/admin/system/*` | superadmin only |

### Block User — Immediate Logout
Blocking deletes the user's Redis session keys (`session:{jti}`) immediately. The blocked user's very next API call returns 403. The Flutter app detects 403 and navigates to login with "Account Blocked". Force-logout is effective within 1 request, not 10 minutes.

## API Conventions

- All endpoints under `/api/v1/`. Never break versioning — add `/api/v2/` for breaking changes.
- **Money**: always integer paise in API payloads (`50000` = ₹500.00). Never use floats.
- **Dates**: stored as UTC, rendered as `Asia/Kolkata` in admin panel responses.
- **Pagination**: cursor-based for large lists (`?cursor=abc&limit=20`). Page-based only for admin reports.
- **Idempotency**: critical POST endpoints (`/recharge/request`, `/withdrawal/request`) accept `Idempotency-Key` header.
- **Error responses**: structured `{"error": {"code": "DUPLICATE_TRANSACTION_ID", "message": "...", "details": {...}, "request_id": "..."}}`.
- **OpenAPI docs** (`/api/docs`, `/api/redoc`): disabled in production, available behind admin auth only.

## Mobile Architecture (Flutter)

The Flutter app (`mobile/`) follows the same Clean Architecture as the backend:

- **Presentation**: Widgets, Pages, BLoC / Riverpod state management
- **Domain**: Entities, use cases, repository interfaces (pure Dart — zero Flutter dependencies)
- **Data**: Remote data source (Dio HTTP), local data source (Hive offline cache), repository implementations

Key libraries: `go_router` (navigation), `get_it` (DI / service locator), `flutter_secure_storage` (JWT token storage), `Hive` (offline cache), `Dio` (HTTP client with auth + retry interceptors).

The `auth_interceptor.dart` automatically attaches the Bearer token and handles 403 → navigate-to-login.

## Business Domain

### Draw Types & Schedule
Four daily draw slots. Betting closes automatically **5 minutes before** each draw time:

| Draw Slot | Draw Types       | Betting Closes |
|-----------|------------------|----------------|
| 1 PM      | Dear-3, Kerala-4 | 12:55 PM       |
| 3 PM      | Dear-3, Kerala-4 | 2:55 PM        |
| 6 PM      | Dear-3, Kerala-4 | 5:55 PM        |
| 8 PM      | Dear-3, Kerala-4 | 7:55 PM        |

- **Dear-3**: 3-digit draw. Bet types: Single, Double, Box.
- **Kerala-4**: 4-digit draw. Same bet types.
- **Jackpot**: Separate lottery type with its own betting rules.
- **Holiday management**: Admin marks dates as holidays — all betting disabled, users see closure notice.

### Result Entry (2-Step Flow)
1. Admin enters winning number → system previews winners + total payout
2. Admin confirms → `settle_draw_result` Celery task credits winners atomically

Result format enforced: Dear-3 = 3 digits only, Kerala-4 = 4 digits only. Wrong format rejected. Once published, results are locked.

### Wallet Rules
- Recharge: max ₹9,999 per request; transaction ID must be numeric, 1–12 digits; duplicate IDs rejected (unique index where status != rejected).
- Recharge is manual UPI — user uploads screenshot, admin approves/rejects.
- Withdrawal: sent to saved bank account (IFSC + account number) or UPI ID — admin approves and enters UTR.
- Admin wallet reset to ₹0 is logged to audit_logs with reason.

### User Registration & Account States
- Phone number + OTP → lands in admin approval queue as `pending`.
- States: `pending` → `active` (admin approves) or `rejected`. Active → `blocked` (admin action).
- Blocked and pending users receive 403 on any authenticated request.

### Admin Audit Log
Every admin action recorded: actor, timestamp, entity, old value, new value. Immutable — no update/delete on `audit_logs`.

### OTP Service
Third-party WhatsApp/SMS gateway (e.g., zplusone.in). Client provides credentials. Adapter lives at `app/infrastructure/external/otp_gateway.py`.

## Key Operational Notes

- **celery-beat must never be scaled** — multiple instances cause duplicate draw-close triggers.
- `settle_draw_result` failures page immediately (Prometheus alert `SettlementTaskFailed`, severity: page).
- `wallet_balance_inconsistency_total > 0` is a critical alert — must remain zero at all times.
- PgBouncer runs in **transaction pool mode** — never use PostgreSQL advisory locks, `LISTEN/NOTIFY`, or `SET LOCAL` in application code. These require session-mode and will fail silently under transaction mode.
- All API containers run as non-root user `sk6` (UID/GID 1001).
- Nginx enforces TLS 1.3-only in production; dev has no TLS, API on `:8000`, Nginx on `:8080`.
